# Git ブランチ・コミット運用ルール

本ドキュメントは、プロジェクトにおけるGitのコミットメッセージ規則（Conventional Commits準拠）と、ブランチの命名・運用規則を定めたものです。

## 1. コミットメッセージのフォーマット
`プレフィックス: [要求ID] コミットの概要`
※要求IDが存在しない軽微な修正の場合は省略可。
※`[VR001ID006-05]` や `([VR001ID006-05])` のように、プレフィックスの直後、または概要の前に明記すること。

### 【絶対順守事項】要求ID単位でのコミット分割
一度の作業で複数の機能追加や仕様書更新（複数の要求IDにまたがる変更）を行った場合、これらを**絶対に1つのコミットにまとめてはいけません**。
必ず対象の要求IDごとにファイルを分けて（または `git add -p` 等を活用して）コミットを分割し、それぞれのコミットメッセージに該当する `[要求ID]` を付与してください。

### プレフィックス一覧（コミット種別）

| プレフィックス | 意味             | 使用するシチュエーション                                 |
| :------------- | :--------------- | :------------------------------------------------------- |
| **ai**         | AI指示・ルール   | AI指示書や開発ルールの追加・更新                         |
| **feat**       | 新機能           | 新しい機能やクラスを追加したとき                         |
| **fix**        | バグ修正         | バグや不具合を修正したとき                               |
| **docs**       | ドキュメント     | 仕様書・マニュアルや、コードのコメントのみを修正したとき |
| **style**      | フォーマット     | 処理に影響しないインデントや改行の修正、命名変更         |
| **refactor**   | リファクタリング | バグ修正や機能追加を伴わない、コードの構造改善           |
| **perf**       | パフォーマンス   | 処理速度やメモリ使用量（GC）の改善を行ったとき           |
| **test**       | テスト           | テストコードの追加や修正                                 |
| **chore**      | 雑務             | ビルド設定、パッケージ追加、`.gitignore`の更新など       |

### コミットメッセージの例
- `feat: [VR001ID001-01] プレイヤーの移動ロジックを追加`
- `docs: [VR001ID002-01] バトル計算式の仕様書を作成`
- `fix: ジャンプ中の接地判定のバグを修正`
- `chore: 汎用ロギングライブラリをプロジェクトに導入`

## 2. ブランチの命名規則と運用

本プロジェクトでは、`master` を常に本番リリース可能な状態に保ち、実際の開発作業の統合はバージョンごとの `develop` ブランチで行うブランチ戦略を厳守します。

### ブランチ構成の基本ツリー
```text
master (常に完成品・リリース可能な状態)
 ┗ (リリース時に develop/vX.Y からマージされ、 `v1.0.0` などのタグが打たれる)

develop/v1.0 (特定のバージョンに向けた開発の主軸・統合ブランチ)
 ┣ feature/VR001ID001-01_対象コレクションの登録・一括機能 (新機能開発)
 ┃ ┗ feat: ...
 ┗ bugfix/issue-12_バグ修正 (バグ修正)
   ┗ fix: ...
```

### ブランチの役割定義

1. **`master` (または `main`)**
   - 常にリリース可能な（バグのない完成した）状態を保つブランチです。
   - 開発作業は行わず、リリース時にのみ `develop` ブランチからマージを受け入れます。
   - バージョンリリース時には、このブランチに対して `v1.0.0` などのGitタグを打ちます。

2. **`develop/vX.Y`（統合・ベースブランチ）**
   - 現在開発中のバージョン（例：`v1.0`）に向けた開発の主軸となるブランチです。
   - 複数の作業ブランチ（feature等）を統合し、テストを行うための環境として機能します。

3. **`feature/...`, `bugfix/...`（作業用ブランチ）**
   - 実際の作業を行う際は、対象バージョンの `develop/vX.Y` ブランチから分岐させて作成します。
   - **機能追加フォーマット**: `feature/{要求ID}_{要求項目}`
     - （例: `feature/VR001ID006-05_入力値のバリデーション`）
   - **バグ修正フォーマット**: `bugfix/{課題・Issue番号}_{修正内容}`
     - （例: `bugfix/issue-12_エクスポート失敗の修正`）
   - 作業完了・テスト後に、元の `develop/vX.Y` へプルリクエスト（またはマージ）を行います。
