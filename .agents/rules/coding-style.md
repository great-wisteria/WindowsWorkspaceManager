# コーディング規約

## 0. 仕様書とプログラムの追跡可能性 (Traceability)
- プログラムを作成・修正する際は、**必ず事前に関連する仕様書を参照**してからコーディングを開始してください。
- 仕様書からコード、コードから仕様書への双方向の追跡を可能にするため、後述の「コメント・ドキュメンテーション」のフォーマットに従い、作成元となった**要求ID**を必ずソースコードに明記してください。

## 1. 変数と公開範囲（カプセル化）
- 外部からアクセスする必要のない変数や関数は、言語の機能（アクセス修飾子やプライベート変数など）を用いて適切にカプセル化（隠蔽）してください。
- 外部クラスから読み取る必要がある値は、直接変数を公開するのではなく、ゲッター（Getter）メソッドやプロパティの仕組みを使用してください。

## 2. パフォーマンスと安全性
- パフォーマンスのボトルネックになりやすい重い処理（例: DOMの頻繁な操作、重い計算処理、ネットワーク通信など）は、適切なキャッシュや非同期化を行ってください。
- マジックナンバー（意味不明な直接の数値）やハードコードされた文字列は避け、定数として定義してください。
- ネスト（インデント）を浅く保つため、早期リターン（Guard Clauses）を積極的に使用してください。

## 3. 命名規則
- プロジェクトの対象言語ごとに、標準的なコーディング規約に厳密に従ってください。また、既存のコードベースの空気を読み、全体の一貫性を保つことが最優先です。
- **Pythonの場合**: PEP 8 に準拠してください（クラス名は `PascalCase`、関数・変数名は `snake_case`）。
- **C#の場合**: Microsoftの公式規約である「C# Coding Conventions」に完全準拠してください。
  - クラス、コンポーネント、関数・メソッド名、プロパティ: `PascalCase`
  - ローカル変数、引数名: `camelCase`
  - スタイル: 中括弧 `{ }` は改行して配置する（Allmanスタイル）など、C#標準に従うこと。
- **C++の場合**: 「Google C++ Style Guide」に完全準拠してください。
  - クラス名、型名、関数・メソッド名: `PascalCase`
  - 変数名、引数名: `snake_case`
  - クラスのメンバ変数: 末尾にアンダースコアを付けた `snake_case_`
- **その他の言語の場合**: その言語の標準スタイル（JavaScript/TypeScriptの標準など）に準拠してください。

## 4. コメント・ドキュメンテーション
- クラス、メソッド、主要な関数の作成時、各項目の上に以下のフォーマットまたは言語標準のドックストリングで必ずコメントを記述してください。
```text
// 機能概要
// 機能概要の説明
// 来歴
// - [要求ID] 機能名 (仕様書と完全一致させること。複数ある場合は列挙する)
// 引数
// 引数名：渡す値の説明(1行程度)
// 戻り値：戻り値の説明(1行程度)
```

## 5. エラーハンドリングとログ出力
- 単なるデバッグ用のログ出力（例: `console.log` や `print`）は本番環境に残さないよう注意してください。
- 致命的なエラーや予期せぬ例外のキャッチ時には、必ずエラー出力用のロガー（例: `console.error` やログライブラリ）を使用し、後から調査が容易になるようにしてください。

## 6. 非同期処理
- 非同期処理を実装する際は、言語標準の最新手法（例: `async / await` や `Promise` など）を積極的に使用し、可読性を高めてください。

## 7. フォルダ構成とモジュール分割
- 汎用的で再利用可能なモジュール設計を心がけてください。
- ソースコードはルートディレクトリ直下には配置せず、必ず `src/` や `components/` のように役割ごとに整理されたディレクトリ配下に配置してください。
- 原則として、**1ファイルにつき1つの主要なクラス（またはコンポーネント）**を定義し、ファイル名とモジュール名の一致など、管理しやすい構造を維持してください。

## 8. テストと動作検証
- 新しいコア機能や複雑なロジックを実装する際は、いきなり本番環境に組み込むのではなく、まずは単体テスト（ユニットテスト）を作成して動作検証を行ってください。
- 確実な動作保証が取れたロジックのみを本番環境へと統合することで、バグの混入を防ぎます。
